Step-by-step guide

Deploy BuildScale to Microsoft Azure

A complete beginner guide to deploying your Node.js + Express website — including the admin panel — to Azure App Service. Follow every step in order and you'll have a live URL in under an hour.

⏱ ~45 minutes ✦ Beginner friendly 💳 Free tier available 🖥 Windows / Mac / Linux
⚠
Read this first
Problems You Will Face on Azure

Yes, you can absolutely deploy BuildScale on Azure — but there are 3 specific problems you must fix first. If you skip these, your site will deploy but the admin panel and forms won't work. Let's go through them.

Problem Why it happens Severity Fix in step
PORT mismatch Azure assigns its own port via process.env.PORT — your server must listen on it Critical Step 1
tickets.json gets wiped Azure's file system is ephemeral — any file your app writes gets deleted on restart or redeploy Critical Step 2
App doesn't start Azure needs to know which file to run — your package.json must have a start script Important Step 3
⚠ The JSON database problem Your current tickets.json file stores all your data. On Azure, every time your app restarts (which can happen automatically), this file gets reset. This means all your tickets and enquiries will be deleted. Step 2 shows you the fix.

01
Fix your code
Update server.js for Azure

Azure assigns a random port number to your app via an environment variable called process.env.PORT. Your server.js already handles this correctly, but let's double-check and also add the trust proxy setting that Azure requires.

Open server.js in VS Code. Find the PORT line and the app.listen line. Make sure they look exactly like this:

server.js — PORT fix
// ✅ This is already in your server.js — just verify it's there const PORT = process.env.PORT || 3000; // ✅ Add this line right after you create the app (after: const app = express()) app.set('trust proxy', 1); // ✅ Your app.listen should use the PORT variable — verify this too app.listen(PORT, () => { console.log(`Server running on port ${PORT}`); });
✅ Your server.js already has this! Looking at your code, const PORT = process.env.PORT || 3000 is already there. Just add the app.set('trust proxy', 1) line right after const app = express();.
02
The most important fix
Solve the Disappearing Data Problem

This is the biggest issue. Azure's file system is not permanent. Your tickets.json will get wiped every time Azure restarts your app (which it does automatically). You have two options:

OptionDifficultyCostRecommended?
Option A
Use Azure Storage (Blob) to save the JSON file		 Medium ~Free Short term
Option B
Switch to Azure Cosmos DB (MongoDB-compatible)		 Medium Free tier Best long term
Option C ← Easiest
Use Azure App Service "Persistent Storage" path /home		 Easy Free ✅ Do this now

Option C — The Easy Fix: Azure App Service on Linux has a special folder called /home that persists across restarts. We just need to tell your app to save tickets.json there instead of in the project folder.

Open server.js and find the DB_FILE line. Change it to this:

server.js — persistent storage fix
// ❌ BEFORE (data gets wiped on restart): const DB_FILE = path.join(__dirname, 'tickets.json'); // ✅ AFTER (data survives restarts on Azure): const DB_FILE = process.env.WEBSITE_INSTANCE_ID ? path.join('/home', 'tickets.json') // Azure Linux persistent storage : path.join(__dirname, 'tickets.json'); // Local development (unchanged)
💡 How this works WEBSITE_INSTANCE_ID is an environment variable that Azure automatically sets. So on Azure it saves to /home/tickets.json (permanent), and on your laptop it saves to your project folder (same as before). No changes to your local workflow.

Then in the Azure Portal (we'll get to this in Step 5), you need to enable Persistent Storage. Go to: App Service → Configuration → General Settings → Persistent Storage → On

03
Tell Azure how to start your app
Add a Start Script to package.json

Azure needs to know which command to run to start your app. Open package.json and update the scripts section to include a start command:

package.json
{ "name": "build_scale", "version": "1.0.0", "main": "server.js", "scripts": { "start": "node server.js", ← ADD THIS LINE "test": "echo \"Error: no test specified\" && exit 1" }, "engines": { "node": ">=18.0.0" ← ADD THIS SECTION TOO }, "dependencies": { "cors": "^2.8.6", "express": "^5.2.1", "uuid": "^13.0.0" } }
⚠ Remove the "stack" dependency Your package.json includes "stack": "^0.1.0" in dependencies, but your server.js doesn't actually use it. Remove it to keep things clean: delete the line "stack": "^0.1.0", from package.json.

Also create a .gitignore file in your project root so you don't accidentally upload node_modules to GitHub (it's huge and unnecessary):

.gitignore — create this new file
node_modules/ .env .azure/ *.log .DS_Store Thumbs.db
04
Azure setup
Create Your Azure Account

If you don't have an Azure account, follow these steps. If you do, skip to Step 5.

Go to azure.microsoft.com and click "Start free"
Sign in with a Microsoft account (or create one — it's free)
Enter a credit/debit card (you won't be charged) — Azure needs it for identity verification on the free tier
Complete the signup. You'll get $200 free credit valid for 30 days + always-free services
You're taken to the Azure Portal at portal.azure.com — this is your control panel
✅ Free tier is fine for BuildScale The F1 Free App Service plan (which we'll use) is completely free with no time limit. It has some limitations (shared CPU, sleeps after inactivity) but is perfect to start. You can upgrade to B1 Basic (~$13/month) when you need always-on performance.
05
Azure Portal
Create Your App Service

App Service is Azure's platform for hosting web apps. Follow these steps exactly in the Azure Portal:

In the Azure Portal, click "Create a resource" (the big + button)
Search for "Web App" and click it, then click Create
Subscription: Leave as default (your free subscription)
Resource Group: Click "Create new" → type buildscale-rg → OK
Name: Type something unique like buildscale-app (this becomes your URL: buildscale-app.azurewebsites.net)
Publish: Select Code
Runtime stack: Select Node 22 LTS (or Node 20 LTS)
Operating System: Select Linux (cheaper and better for Node.js)
Region: Choose Central India or South India (closest to you in Bareilly)
Pricing Plan: Click "Create new" → name it buildscale-plan → choose F1 Free
Click "Review + Create" → then "Create" — wait 2–3 minutes
When done, click "Go to resource" — you're now in your App Service

Now enable Persistent Storage so your tickets.json survives restarts:

In your App Service, click Configuration in the left sidebar
Click the General settings tab
Scroll down to find "Persistent storage" and toggle it On
Click Save at the top → click Continue in the confirmation popup
06
Version control
Push Your Code to GitHub

Azure will deploy directly from GitHub. Open a terminal (in VS Code: press Ctrl + `) and run these commands one by one from inside your project folder:

Terminal — run these commands in order
# 1. Initialize git in your project folder (only do this once) $ git init # 2. Stage all your files for commit $ git add . # 3. Create your first commit $ git commit -m "Initial BuildScale deployment" # 4. Go to github.com → New Repository → name it "buildscale" → Create # Copy the repository URL (looks like: https://github.com/yourusername/buildscale.git) # Then run this (replace the URL with yours): $ git remote add origin https://github.com/YOUR_USERNAME/buildscale.git # 5. Push your code to GitHub $ git branch -M main $ git push -u origin main
💡 First time using Git? When you run git push, it may ask for your GitHub username and password. For the password, you need to use a Personal Access Token (not your GitHub password). Go to GitHub → Settings → Developer Settings → Personal Access Tokens → Generate new token → check "repo" scope → copy the token and use it as the password.
07
The big moment
Connect GitHub to Azure & Deploy

Now we connect your GitHub repository to Azure so it automatically deploys every time you push code.

In your App Service on Azure Portal, click Deployment Center in the left sidebar
Under Source, select GitHub
Click Authorize and sign in to your GitHub account
Organization: Select your GitHub username
Repository: Select buildscale
Branch: Select main
Click Save at the top
Azure automatically triggers the first deployment — watch the Logs tab to see progress
Wait 3–5 minutes for deployment to complete (you'll see a ✅ green checkmark)
Click the URL at the top of your App Service (something like buildscale-app.azurewebsites.net) — your site is live! 🎉
🚀 Auto-deploy is now set up! From now on, every time you run git push from your laptop, Azure will automatically redeploy your site within 2–3 minutes. No manual steps needed.
08
Most important for you
Accessing the Admin Panel on Azure

Your admin panel is built into your Express server and is served at the /admin route. Once deployed, you access it at:

Your admin panel URLs (replace with your actual app name)
# Admin dashboard (tickets + enquiries) https://buildscale-app.azurewebsites.net/admin # All tickets as JSON https://buildscale-app.azurewebsites.net/api/tickets # All enquiries as JSON https://buildscale-app.azurewebsites.net/api/enquiries # Dashboard stats https://buildscale-app.azurewebsites.net/api/stats
🔒 Security Warning — Protect Your Admin Panel Right now, anyone who knows the URL can access your admin panel. Before going live, you should add password protection. Here's the simplest way — add this to your server.js:
server.js — add basic admin password protection
// Add this BEFORE your app.get('/admin', ...) route // Simple admin password middleware function adminAuth(req, res, next) { const adminPassword = process.env.ADMIN_PASSWORD || 'changeme123'; const auth = req.headers.authorization; if (!auth || !auth.startsWith('Basic ')) { res.set('WWW-Authenticate', 'Basic realm="BuildScale Admin"'); return res.status(401).send('Unauthorized'); } const credentials = Buffer.from(auth.slice(6), 'base64').toString(); const [user, pass] = credentials.split(':'); if (user === 'admin' && pass === adminPassword) { return next(); } res.set('WWW-Authenticate', 'Basic realm="BuildScale Admin"'); res.status(401).send('Wrong credentials'); } // Now protect the admin routes app.get('/admin', adminAuth, (req, res) => { res.send(getAdminHTML()); }); app.get('/api/tickets', adminAuth, (req, res) => { /* existing code */ }); app.get('/api/enquiries', adminAuth, (req, res) => { /* existing code */ });

Then set your password securely in Azure (never hardcode it). In the Azure Portal:

Go to your App Service → Configuration → Application settings
Click + New application setting
Name: ADMIN_PASSWORD — Value: your chosen password (make it strong!)
Click OK then Save
💡 How login works When you visit /admin, your browser will pop up a login dialog. Enter username: admin and the password you set in Azure settings. Your browser will remember it for the session.

Optional but recommended
Connect a Custom Domain

Right now your site is at something.azurewebsites.net. To use your own domain (like buildscale.io), follow these steps:

Buy a domain from GoDaddy, Namecheap, Google Domains, or any registrar
In Azure App Service → Custom domains → + Add custom domain
Enter your domain (e.g. www.buildscale.io) → click Validate
Azure shows you DNS records to add. Go to your domain registrar and add them (a CNAME and TXT record)
Wait 15–60 minutes for DNS to propagate, then click Add custom domain
Enable HTTPS / SSL: In Custom domains → your domain → Add binding → App Service Managed Certificate (free!)
⚠ Custom domains need at least B1 plan The free F1 plan does not support custom domains. You'll need to upgrade to at least B1 Basic (~₹1,100/month) to use your own domain. The azurewebsites.net URL works fine on the free plan.

When things go wrong
Common Problems & Fixes
ProblemCauseFix
Site shows "Application Error" or 500 App crashed on startup Go to App Service → Log stream (left sidebar) to see the error in real time
"Cannot find module" error node_modules not installed Make sure package.json has a "start" script. Azure runs npm install automatically if it sees package.json
Form submissions return 404 or network error API routes not found Check that your HTML files are calling the correct API URL. On Azure they should use relative paths like /api/tickets (not localhost)
Site works but admin shows blank JavaScript error in admin HTML Open browser DevTools (F12) → Console tab to see errors
Tickets disappear after a day Persistent Storage not enabled Go back to Step 2 and enable Persistent Storage in Azure Portal
Deployment succeeded but site is old Browser cache Hard refresh: Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac)
App sleeps and takes 10–20s to load F1 free plan goes to sleep after inactivity Normal on F1. Upgrade to B1 for always-on, or use a free service like UptimeRobot to ping your site every 5 minutes

To view your live logs anytime, run this in your terminal (with Azure CLI installed):

Terminal — view live logs
# Install Azure CLI first: https://docs.microsoft.com/en-us/cli/azure/install-azure-cli $ az login # Then stream live logs (replace with your names) $ az webapp log tail \ --resource-group buildscale-rg \ --name buildscale-app

Or view logs directly in the Azure Portal: App Service → Log stream in the left sidebar. You'll see every console.log() from your server.js in real time.

How to redeploy after making changes:

Terminal — update your live site (run every time you make changes)
# After making any changes to your code: $ git add . $ git commit -m "describe what you changed" $ git push # Azure automatically redeploys in 2-3 minutes. That's it!
